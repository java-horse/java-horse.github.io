MySQL索引优化简析

索引分类

单值索引

概念：一个索引只包含单个数量的列，一个表可以有多个单值索引

唯一索引

索引列的值必须唯一，但是允许有空值存在

主键索引

字段被设置为主键后，数据库会自动为该字段建立索引，innodb引擎下被称为聚簇索引

复合索引

一个索引包含多个列，一个表中索引的数量尽量不要太多

基本语法

索引创建时机

适合创建索引：

  • 频繁作为查询条件的字段应该创建索引
  • 查询中与其他表关联的字段适合创建索引（外键关系字段）
  • 查询中作为排序条件的字段适合创建索引（排序字段创建索引后能够大幅度体生排序速度）
  • 查询中作为统计或者分组的字段

注意：一般来说，复合索引的性价比要比单值索引的性价比要高，如果条件允许情况下，尽量选择复合索引

不适合创建：

  • 表数据过少
  • 经常增删改的表或者字段
  • where条件中用不太到的字段
  • 过滤性不好的字段（字段值的辨识性不高，例如：用户的性别：男/女，文章类型：原创/转载 等等）

关键字简析

id

查询的序列号，包含一组数字，表示查询中执行select子句或者操作表的顺序

  • id值相同：从上至下一次执行
  • id值不同：如果是存在子查询，id值的序号会递增，id值越大优先级越高，越先被执行
  • id值同时存在相同和不同：id如果相同，可认为是一组，从上至下顺序执行；在所有组中，id值越大，优先级越高

select_type

查询的类型，主要用于区别普通查询、联合查询、子查询等复杂查询

类型图示：

table

表示数据来源于那张表（没什么实际作用）

type

查询的访问类型，比较重要的优化指标，指标的值从最好到最坏依次如下所示：

  1. system
  2. const
  3. eq_ref
  4. ref
  5. fulltext
  6. ref_or_null
  7. index_merge
  8. unique_subquery
  9. index_subquery
  10. range
  11. index
  12. all

注意：一般来说，得保证查询至少达到 range 级别，最好能达到 ref 级别

possible_keys

显示查询过程中可能应用在这张表中的索引，存在一个或者多个。

注意：查询涉及到的字段上存在索引，则该索引被列出，但是不一定被查询实际使用。

key

查询中实际使用到的索引，如果为null,则表示没有使用索引

ken_len

表示索引中使用的字节数，可通过该列计算查询中使用到的索引的长度。

含义：key_len字段能够检查此次查询是否充分利用上了索引。key_len的值越大，表明该索引使用的越充分。

索引计算说明：

ref

显示索引的那一列被使用了。如果可能的话，此值显示为一个常数，表明那些列或者常量被用于查找索引列上的值。

rows

显示MySQL认为此查询语句执行查询时必须检查的行数，越少越好。

extra

其他的额外重要的信息

  • using where：表明使用了 where 条件过滤
  • using firesort：MySQL会对数据使用一个外部的索引排序，而不是按照表内的索引排序。无法利用索引完成的排序操作被称为“文件排序”
  • using temporary：MySQL在对查询结果排序时会使用临时表保存结果，常见于 order by 排序和 group by 的分组操作中
  • using index：代表表示相应的 select 操作中使用了覆盖索引(Covering Index)，避免访问了表的数据行，效率不错！ 如果同时出现 using where，表明索引被用来执行索引键值的查找;如果没有同时出现 using where，表明索引只是 用来读取数据而非利用索引执行查找
  • using join buffer：表示使用了连接缓存

索引失效简析

全值匹配我最爱

解释：where后面需要过滤的字段按照顺序（不按照顺序也可以，SQL会自动优化）在索引中可以匹配的上

最左前缀要遵守

解释：

  • 带头大哥不能丢：一定要按照建立索引的字段顺序进行where后面过滤条件的书写，带头的索引字段需要写在最前面，否则后面字段的索引全部失效（全部失效）
  • 中间兄弟不能断：过滤条件的书写不要字段，一旦跳过某个字段，那么此字段后面的索引字段都将失效（部分失效）

索引列上少计算

不要再索引列上进行【计算，函数，手动/自动类型转换】操作，否则索引将全部失效

范围之后全失效

如果在复合索引的某个字段上使用范围查询，那么后面字段的索引将全部失效

建议：建立索引时，将可能会用到范围查询的字段尽量放到

LIKE百分写最右

只有 % 匹配符写在右边的时候，索引才不会失效。类似于 %blog%，%blog 这样的匹配索引都将失效。

覆盖索引不写*

查询字段的时候，如果这些字段全部都存在于索引中，一定不要写 * 符号，尽量按照字段进行查询。

不等空值还有OR

不等空值还有OR，隐式转换要注意。

  • 不等号（!=,<>）：MySQL在使用不等条件过滤时，有时索引会失效，导致全表扫描
  • is not null索引失效，全表扫描 is null索引失效
  • 使用 or 作为过滤条件，有时也会导致索引失效，全表扫描
    • 建议：如果条件允许情况下，可以使用 union all 或者 union 来替代 or
  • 防止值类型的隐式转换，例如：char 类型隐式转换为 int 类型，会导致索引失效

索引优化简析

关联查询优化

left join：

  • 优化关联查询时，只有在被驱动表上创建索引才能生效
  • left join 时，左侧的为驱动表，右侧为被驱动表

inner join：

  • inner join 时，MySQL会自动优化，将小结果集的表作为驱动表

注意：能够直接多表关联的尽量直接使用 join 关联，不要使用子查询（很难使用到索引）

排序分组优化

  1. order by 当没有 where 或者 limit 过滤条件时，不会使用索引
  2. order by 的两个索引字段排序时，书写的顺序不能乱，例如：索引顺序：a-b，那么order by 的顺序如果是：b-a，必将会进行文件排序：user fielsort
  3. group by 没有过滤条件时，也会使用到索引
